## 研究背景

本项目是一个 **Canvas LMS 教学助手 Bot**，核心功能包括：

- Canvas 课程数据自动同步（作业、公告、成绩）
- 定时主动推送（作业到期提醒、成绩更新通知）
- AI 对话（基于 LangChain + RAG 的课程知识问答）
- 课程绑定（OAuth2 授权流程）

**技术架构优势**：业务逻辑层与消息通道层已解耦（FastAPI 后端 + 消息路由），新增渠道仅需实现消息适配器，核心逻辑无需改动。

## Tier 1：最优先考虑的渠道

### 1. Telegram Bot + Mini App

#### 平台概况

Telegram 是全球增长速度最快的即时通讯平台之一。截至 2025 年 3 月，月活用户突破 **10 亿**，其 Mini App 生态正处于爆发期——已有超过 75,000 个活跃 Mini App，年增长率 200%。

#### 为什么是最优选择

- **零审核门槛**：无需应用商店审核，代码部署即上线，极大降低冷启动成本
- **全 WebView 能力**：Telegram Mini App 本质是在 WebView 中运行的 HTML5 应用，任何 Web Canvas 库（Fabric.js、Konva.js、Three.js）均可直接使用
- **原生 Bot API 完善**：主动推送消息、内联键盘交互、回调查询、文件/图片发送均为标准能力
- **变现基础设施成熟**：Telegram Stars 虚拟货币体系（2025-2026年新能力），用户通过 Apple/Google Pay 或 TON 加密货币购买 Stars，开发者可兑换收益
- **技术迁移成本最低**：项目后端（FastAPI）可直接对接 Telegram Bot API，消息适配器开发量极小

#### 目标用户分析

- **留学生群体**：海外中国留学生高度依赖 Telegram 作为日常通讯工具，且普遍使用 Canvas LMS
- **国际学校**：全球约 13,000 所国际学校使用 Canvas，其师生群体是 Telegram 的活跃用户
- **先发优势**：当前 Telegram 生态内 **几乎没有** Canvas LMS 教学助手类应用，蓝海市场

#### 技术实现要点

```
用户侧                   Telegram 侧                    本项目后端
───────                ─────────────                  ──────────
Telegram App  ──────▶  Bot API (Webhook)  ──────▶  FastAPI (已有)
   │                                                   │
   └── Mini App (WebView) ────────────────────────▶  Canvas OAuth2
                                                      AI Chat (RAG)
                                                      Data Sync (Celery)
```

- 复用现有 FastAPI 路由，新增 `/telegram/webhook` 端点
- Mini App 内嵌 WebView 承载 Canvas 授权页面和交互界面
- Telegram Bot API 的 `sendMessage` / `editMessageReplyMarkup` 替代 iLink Bot API 调用

#### 风险与注意事项

- **Sybil 攻击**：Telegram 生态存在大量机器人农场（某些游戏中 ~80% 用户为 farming bot），需做好用户验证
- **TON 区块链依赖**：自 2025 年 1 月起，Telegram Mini App 的加密支付仅支持 TON 链
- **国内网络限制**：中国大陆用户访问 Telegram 需翻墙，无法覆盖国内高校主力用户

### 2. 钉钉机器人 + AI Agent

#### 平台概况

钉钉当前拥有 **8 亿注册用户**，覆盖 **2,600 万+ 企业组织**，在中国企业协作市场份额超过 30%。2025 年钉钉全面转型为"AI钉钉"，推出 Agent OS 战略，平台上已有 **141 万** AI 应用。

#### 为什么值得投入

- **教育机构渗透率极高**：大量中小学、高校、教培机构使用钉钉进行教学管理
- **AI Agent 生态红利**：钉钉 2025-2026 年大力推动 AI 应用生态，给予开发者流量和算力扶持
- **企业付费意愿强**：钉钉上的 B2B SaaS 订阅收入在 2025 财年突破 **300 亿人民币**
- **Canvas 类工具有刚需**：白板、作业批改可视化、学情数据看板在钉钉生态内属于高频场景

#### 技术实现要点

- 通过钉钉开放平台 (`open.dingtalk.com`) 注册机器人应用
- 接入钉钉 AI Agent SDK，将现有的 LangChain Agent 封装为钉钉 AI 插件
- 利用钉钉宜搭（低代码平台）快速构建教学数据看板
- 钉钉卡片消息格式支持 Markdown + 按钮交互，适配现有消息模板

#### 目标用户

- 国内 K12 学校教师和学生
- 教培机构的教学管理
- 高校行政/教务系统集成

### 3. Tauri 桌面应用 + PWA

#### 平台概况

| 指标 | Tauri v2 | Electron | Flutter |
|------|----------|----------|---------|
| 安装包大小 | **2-5MB** | ~150MB | ~15MB |
| 启动时间 | **<0.2s** | 2-5s | ~1s |
| 空闲内存 | **30-50MB** | 200-400MB | 100-200MB |
| 平台覆盖 | 桌面 + 移动端 | 仅桌面 | 全平台 |
| 开发语言 | JS前端 + Rust后端 | JS/TS + Node | Dart |
| GitHub Stars | 106k+ | 115k+ | 167k+ |

#### 为什么需要独立应用

- **摆脱平台依赖**：不受任何平台 API 变更、审核、封禁影响，是长期战略保障
- **Tauri v2 技术成熟**：40x 小于 Electron，10x 更少内存，移动端（iOS/Android）支持已稳定
- **PWA 作为 Web 兜底**：零安装、零审核，用户浏览器直接访问，适合不愿意安装 App 的用户
- **完整 Canvas 能力**：任何 Web 渲染引擎（Chromium WebView）提供完整的 HTML5 Canvas 支持

#### 路线建议

- **PWA 先行**：将现有 Web 界面升级为 PWA（离线缓存 + 推送通知），成本极低
- **Tauri 跟进**：基于 PWA 版本打包为 Tauri 桌面应用，获得原生体验 + 系统托盘提醒
- **工具推荐**：可使用 PakePlus 将 Web 版本快速打包为 <5MB 桌面应用

## Tier 2：值得探索的渠道

### 4. 飞书 Bot

#### 优势

- **互动卡片系统是调研中最优秀的 Bot 消息格式**：支持复杂布局、条件渲染、表单输入、按钮回调，远超微信 iLink 的 Markdown 卡片
- **飞书多维表格** (Bitable) 可作为课程数据看板，通过 API 将 Canvas 同步数据写入
- **飞书 Canvas 文档**：原生协作白板，可嵌入教学可视化内容
- 用户集中在互联网/科技公司（员工培训 L&D 场景需求明确）

#### 局限

- 用户体量远小于钉钉（尤其在教育行业）
- 需要企业认证，个人开发者接入门槛较高

#### 适配场景

- 面向科技公司的企业培训（Learning & Development）
- Canvas 课程数据实时同步到飞书多维表格
- 在飞书 Canvas 文档中嵌入 AI 教学助手

---

### 5. 企业微信

#### 优势

- **独特优势**与微信生态直连：可同时服务微信用户和企业微信用户
- 2025 年公布的 **6 种 AI Agent 集成路径**，生态日益开放
- 适合面向家长的**家校沟通**场景（学校使用企业微信管理，家长使用微信接收）

#### ⚠️ 关键限制

- 对**外部联系人（客户）的主动群发消息无公开 API**——这意味着无法像微信 iLink Bot 一样向学生主动推送作业提醒
- 内部机器人可行，但面向客户（学生/家长）的自动化需要切换到"微信客服"产品线
- 智能机器人（Smart Bot）目前没有公开 API

#### 适配场景

- 学校内部教务管理（面向教职工的内部 Bot）
- 与微信 iLink Bot 组合使用（微信负责学生触达，企业微信负责教师管理端）

### 6. QQ 机器人

#### 优势

- QQ 用户 550M+ MAU，**年轻用户占比较高，精准命中高校学生群体**
- 技术迁移成本最低：QQ Bot API 与 iLink Bot API 结构相似，消息适配器开发量极小
- 支持群聊、频道（QQ频道）、私聊多种场景

#### ⚠️ 重大限制（评估后不推荐作为主渠道）

| 限制项 | 详情 | 对本项目的影响 |
|--------|------|---------------|
| **主动消息封禁** | Bot 不能再主动向群发送消息，只能被 @ 后被动回复 | **致命**：定时推送作业提醒是本项目核心功能 |
| **DAU 门槛** | Ark 卡片 ≥ 400 DAU；Markdown 模板 ≥ 2,000 DAU；高级 Markdown ≥ 10,000 DAU | 冷启动阶段无法使用丰富的消息格式 |
| **审核周期** | 2-3 天审核，最多 24 个斜杠指令 | 迭代速度受限 |
| **Markdown 限制** | 图片 URL 需白名单；不支持图文混排 | 消息呈现能力大打折扣 |

#### 结论

QQ Bot 可作为**微信的辅助渠道**（复用消息适配器），但**不能替代微信的主动推送能力**。建议在微信版成熟后，用 QQ Bot 承接被动查询类需求。

### 8. 其他小程序生态

#### 支付宝小程序

- **6.44B MAU**，API 覆盖率为微信的 41.5%
- 偏金融/政务场景，教育类应用生态薄弱
- **建议**：如果 Canvas 产品涉及课程缴费/支付场景，可考虑用 Taro 顺手编译一版

#### 抖音小程序

- **2.73B MAU**，API 覆盖率为微信的 27.5%
- 内容电商生态强大，但教学工具型应用冷启动困难
- **建议**：仅当产品形态适配短视频（如"每日一题"视频互动）时考虑

#### 百度智能小程序

- API 覆盖有限，Canvas 支持弱
- 唯一优势是搜索 SEO 流量分发
- **建议**：优先级最低，可用 Taro 顺手编译

#### 跨平台策略

使用 **Taro** 或 **uni-app** 框架，一套代码可同时编译到微信/支付宝/抖音/百度/QQ 五大小程序平台：

- 开发成本降低 40-60%
- 上线周期缩短至 2-4 周（vs 每平台 2-3 月）
- **局限性**：各平台 API 交集仅约 24%，Canvas 差异较大，需针对性适配

## 关键结论

### 1. 主动推送是不可妥协的核心能力

本项目依赖定时主动推送（作业到期提醒、成绩更新通知、公告同步），因此：
- **QQ Bot 不适合**作为主渠道（已封禁主动消息）
- 企业微信的**外部联系人主动群发无 API**，同样受限
- Telegram、钉钉、飞书均完整支持主动推送

### 2. 项目架构已具备多渠道扩展基础

README 明确提到的"业务逻辑与消息层解耦"是当前最大的技术优势。新增渠道仅需实现对应的消息适配器（Message Adapter），核心的同步引擎、AI Agent、OAuth 流程均可复用。

### 3. Telegram Bot 是近期最优的新渠道选择

理由：零审核 + 1B 用户 + 全 WebView 能力 + 原生变现 + 留学生精准覆盖 + 赛道空白。开发成本低，验证速度快。

### 4. 独立应用（Tauri + PWA）是长期战略保障
